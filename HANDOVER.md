## 1. WHAT THIS PROJECT IS

FutureTokenization.com started (March 2026, hand-built by Khurram) as a single-page **educational codex** — a 2,371-line `index.jsx` covering tokenization, RWA, stablecoins, CBDCs, tri-lingual (EN/AR/FR), with a "Digital Czar" chatbot.

**The pivot (25 Jun 2026, Khurram's decision):** make it *the link between investors and every VARA-licensed virtual-asset provider in the UAE*. Model **A+B**: a live, verified directory now (Model A); pay-per-lead introductions later (Model B — **legal-gated**: "arranging/introducing" can itself be VARA-regulated activity; requires UAE lawyer sign-off before building). The site stays a discovery/education platform, **not** a broker or advisor. Education remains first-class — a redesign that demoted the codex was explicitly rejected (see §9).

The product's moat is **data integrity**: every displayed register number traces to VARA's own register, verified programmatically, with absent/unverifiable data shown as absent — never faked (NO PHANTOM READINGS).

## 5. THE DATA SPINE & REGISTER PIPELINE (the moat)

**Source of truth:** VARA public register. The site is Gatsby+Umbraco: the full register lives in Gatsby static-query JSON — get hashes from `/page-data/en/licenses-and-register/public-register/page-data.json`, then fetch `/page-data/sq/d/<hash>.json`; collect `contentTypeAlias == 'varaRegistry'`.

**Re-pull procedure (scripted, run in repo root):**
```bash
python3 scripts/pull-vara-register.py        # refuses to WRITE unless Active VL refs exactly
                                             # match the server-rendered register page
python3 scripts/sync-corpus-register.py      # regenerates corpus PART 2 table + activity counts
python3 scripts/gen-czar-corpus.py           # rebuilds lib/czarCorpus.js
# then manually: update the count in pages/api/chat.js owner directive + corpus Part 1
# count lines if the headline moved; npx next build; commit+push; curl-verify live.
```

**Known data quirks (all handled in the script):**
- `VL/23/09/001` is genuinely shared by TWO real firms (BitGo MENA + CoinMENA) — 51 firms ↔ 50 refs. Dedupe keeps both (equal-status, different-name exception).
- CMS urls use `/vara-en/` prefix (normalize to `/en/`). `licenseActivities` arrives as a list. CMS "(N)" duplicate nodes dropped. Stale duplicate-ref nodes resolved by status rank (Active/Issued > Withdrawn > Pending — how the WadzPay Pending-vs-Withdrawn pair resolves).
- Some refs use 4-digit years (`VL/2024/12/001`) — regexes must accept `\d{2,4}`.
- IPA table is NOT server-rendered on the register page — it only exists in the static-query JSON; per-entity detail pages ARE server-rendered (use them for spot verification).
- Local Homebrew Python has no SSL roots — the script shells out to `curl`.

**Integrity incidents log (why the paranoia is earned):**
1. WebFetch fabricated register entries (Jun 25).
2. Claude's own parser under-counted 49→47 via an empty-field assumption (Jun 25).
3. This session's first pull script draft **merged BitGo+CoinMENA** (shared-ref) and mis-kept WadzPay's stale "Pending" node — both caught by self-check before shipping. The cross-check-vs-rendered-page gate refused to write twice before the data was right.
